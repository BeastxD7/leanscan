# Database

Schema + standalone Postgres for local dev.

## What's here

| File | Purpose |
|---|---|
| `Dockerfile` | Custom Postgres 16 Alpine image with LeanScan credentials baked in |
| `docker-compose.yml` | Spins up Postgres exposed on host port `54320` |
| `migrations/` | **Historical** — original raw SQL migrations from before Prisma. Not used by any tooling now. |

The active schema source of truth is `coding/api/prisma/schema.prisma`. Migrations are generated by Prisma into `coding/api/prisma/migrations/`.

## Start Postgres

```bash
cd coding/db
docker compose up -d
```

Verify it's healthy:

```bash
docker compose ps
# postgres should show "Up X seconds (healthy)"
```

Connect with `psql` from your host (to debug or browse):

```bash
psql -h localhost -p 54320 -U leanscan -d leanscan_dev
# password: dev
```

Or with a GUI like TablePlus / DBeaver / Postico:
- Host: `localhost`
- Port: `54320`
- User: `leanscan`
- Password: `dev`
- Database: `leanscan_dev`

## Stop Postgres

```bash
cd coding/db
docker compose down            # keeps data
docker compose down -v         # wipes all data (DANGEROUS)
```

## After Postgres is up, run migrations

From the API folder, not here:

```bash
cd ../api
npm run migrate:dev
```

Prisma generates and applies any pending migrations.

## When to wipe and start over

```bash
cd coding/db
docker compose down -v
docker compose up -d
cd ../api && npm run migrate:dev -- --name init
```

This is the nuclear reset — useful when the schema changes drastically or you want a clean slate for testing signup flows etc.

## Why a custom Dockerfile instead of just `postgres:16-alpine` directly

Two reasons:
- Credentials baked into the image so anyone cloning the repo just runs `docker compose up` with no extra env to configure.
- Anywhere we deploy this image (dev machine, CI, staging) we get the same credentials, same locale, same init args. No surprises.

Production (Phase 7) doesn't use this image — production runs Supabase Postgres which we treat as a managed service.
